> Released January 22, 2025. Quick map of headline changes, breaking behavior, and the upgrade items most likely to bite. Deep-link to existing notes when the topic has its own page.

## At a glance

| Change                                                              | Type            | Impact                                                                                                                   |
| ------------------------------------------------------------------- | --------------- | ------------------------------------------------------------------------------------------------------------------------ |
| Node.js 20+ required                                                | Breaking        | Drop Node 16/18 from CI before bumping                                                                                   |
| Express **v5** is the default                                       | Breaking        | Wildcard routes (`*`, `(.*)`) need names: `*splat`                                                                       |
| Fastify **v5** under `@nestjs/platform-fastify`                     | Breaking        | CORS no longer auto-allows `PUT/PATCH/DELETE`; [[nestjs/fundamentals/middleware\|middleware]] paths need named wildcards |
| `CacheModule` on `cache-manager` v6 + [[nestjs/data/caching\|Keyv]] | Breaking        | Old `redisStore` config removed; see [[nestjs/data/caching\|the caching recipe]]                                         |
| `ConfigService#get` lookup order changed                            | Breaking        | Internal config now overrides `process.env`                                                                              |
| Termination hook order **reversed**                                 | Breaking        | `OnModuleDestroy` etc. now run reverse-init                                                                              |
| Module resolution uses object refs, not metadata hashes             | Breaking (edge) | Same dynamic import twice = two instances unless you share a variable                                                    |
| Global-module middleware runs first                                 | Breaking (edge) | Global middleware order is now deterministic                                                                             |
| `Reflector#getAllAndOverride` returns `T \| undefined`              | Type-only       | Handle the undefined branch                                                                                              |
| `ConsoleLogger` JSON mode                                           | Feature         | [`new ConsoleLogger({ json: true })`](#consolelogger-json-mode)                                                          |
| Microservice `unwrap()`, `on()`, `status`                           | Feature         | Native client access + connection events                                                                                 |
| `ParseDatePipe` in `@nestjs/common`                                 | Feature         | Built-in date parsing                                                                                                    |
| Microservice options from the DI container                          | Feature         | `useFactory`/`inject` work for transporter config                                                                        |
| `@nestjs/cqrs`: request-scoped + typed handlers                     | Feature         | Generic `Command<R>` / `Query<R>` infer the result type                                                                  |
| `IntrinsicException`                                                | Feature         | Throw without auto-logging                                                                                               |
| Node.js startup is faster                                           | Performance     | Free; large dynamic-module apps benefit most                                                                             |

## Breaking changes you'll actually hit

### Node.js 20+

NestJS 11 drops Node 16 ([EOL Sept 2023](https://nodejs.org/en/about/previous-releases)) and Node 18 ([EOL April 2025](https://nodejs.org/en/about/previous-releases)). Bump your Dockerfile, CI matrix, and `engines` field together.

```json
// package.json
"engines": { "node": ">=20" }
```

### Express v5: wildcards must be named

Every `*` route now needs a name. Pre-v11 patterns still resolve via Nest's compatibility layer, but they emit a warning at startup and the migration is mechanical:

```typescript
import { Controller, Get } from "@nestjs/common";

@Controller()
export class UsersController {
  // Before (Express v4): still works in v11, but Nest auto-rewrites it
  // to a valid Express v5 route and logs a warning. Don't ship new code
  // with this shape.
  @Get("users/*")
  findAllV4() {
    return "auto-converted, deprecated";
  }

  // After (Express v5)
  @Get("users/*splat") // matches /users/anything (not /users alone)
  findAll() {}

  // Or, including the bare path
  @Get("users/{*splat}") // matches /users AND /users/anything
  findAllIncludingBase() {}
}
```

Same fix for [[nestjs/fundamentals/middleware|middleware]] route registration:

```typescript
import { Module, NestModule, MiddlewareConsumer } from "@nestjs/common";
import { MyMiddleware } from "./my.middleware";

@Module({})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    // Before:  consumer.apply(MyMiddleware).forRoutes("*")
    consumer.apply(MyMiddleware).forRoutes("{*splat}");
  }
}
```

`splat` has no special meaning; pick any identifier. Other Express v5 path-string changes: `?` is no longer optional (use `/:file{.:ext}`), regex chars aren't supported, and `()[]?+!` are now reserved (escape with `\`).

### Express v5: `qs` query parser is gone

Express 5 ships with the simple query parser by default; the `qs` parser that powered nested-bracket query strings (`?filter[where][name]=John`, `?item[]=1&item[]=2`) is no longer the default ([Express 5 migration notes](https://expressjs.com/en/guide/migrating-5.html)). To restore v4 behavior:

```typescript
// main.ts
import { NestFactory } from "@nestjs/core";
import { NestExpressApplication } from "@nestjs/platform-express";
import { AppModule } from "./app.module";

async function bootstrap() {
  const app = await NestFactory.create<NestExpressApplication>(AppModule);
  app.set("query parser", "extended");
  await app.listen(3000);
}
bootstrap();
```

Note the `<NestExpressApplication>` generic; `app.set()` only exists on the Express adapter.

### Fastify v5: CORS allow-list shrank

Fastify v5's CORS plugin now requires the allowed methods to be specified explicitly when you need anything beyond the [CORS-safelisted methods](https://fetch.spec.whatwg.org/#cors-safelisted-method) (`GET`, `HEAD`, `POST`); see [`@fastify/cors` README → Options](https://github.com/fastify/fastify-cors#options). Add `PUT`, `PATCH`, `DELETE` if your API uses them:

```typescript
import { NestFactory } from "@nestjs/core";
import { NestFastifyApplication, FastifyAdapter } from "@nestjs/platform-fastify";
import { AppModule } from "./app.module";

const app = await NestFactory.create<NestFastifyApplication>(AppModule, new FastifyAdapter(), {
  cors: { methods: ["GET", "POST", "PUT", "PATCH", "DELETE"] },
});
```

### `CacheModule` ↔ Keyv

The pre-v3 `store: redisStore({...})` config is gone. Pass [Keyv](https://keyv.org/docs/) stores via the `stores` array instead. Full setup, multi-tier example, and the `value`/`expires` wire-format change live in [[nestjs/data/caching|the caching recipe]].

> [!warning] Wire format changed
> Cached values are now stored as `{ "value": ..., "expires": <ms epoch> }` ([Keyv `set()` builds `{ value, expires }` from `calculateExpires(ttl)` then encodes](https://github.com/jaredwray/keyv/blob/main/core/keyv/src/keyv.ts); [`calculateExpires`](https://github.com/jaredwray/keyv/blob/main/core/keyv/src/utils.ts) returns `Date.now() + ttl` for a positive finite TTL in ms, otherwise `undefined`). Reading via the `cache-manager` API hides the wrapper, but raw Redis access (a `redis-cli GET` from outside Nest) returns the wrapped object. Migrating an existing cache: either flush it on deploy or write a one-off transform. Source: [official migration guide](https://docs.nestjs.com/migration-guide#cache-module).

### `ConfigService#get` lookup order

In [[nestjs/releases/v10|v10]], validated env vars and `process.env` were read **before** internal configuration could override them; in v11 the order is reversed: **custom config wins**, then validated env, then `process.env` ([migration guide → Config module](https://github.com/nestjs/docs.nestjs.com/blob/master/content/migration.md#config-module)). If you used to override a value by exporting `FOO=...` before starting the app, that override is now invisible whenever a config factory returns the same key.

`ignoreEnvVars` is [deprecated in favor of `validatePredefined`](https://github.com/nestjs/config/blob/master/lib/interfaces/config-module-options.interface.ts#L28-L61) (set `validatePredefined: false` for the same effect). The new `skipProcessEnv: true` opt-out makes `ConfigService` ignore `process.env` entirely (useful for tests). See [@nestjs/config README](https://github.com/nestjs/config#readme).

### Termination hook order is reversed

For modules wired `A → B → C` (A depends on B depends on C):

| Phase                                                                             | v10 order | v11 order             |
| --------------------------------------------------------------------------------- | --------- | --------------------- |
| Init (`OnModuleInit`, `OnApplicationBootstrap`)                                   | C → B → A | C → B → A (unchanged) |
| Destroy (`OnModuleDestroy`, `BeforeApplicationShutdown`, `OnApplicationShutdown`) | C → B → A | **A → B → C**         |

This is the correct order: dependents shut down before their dependencies, so a service can still call its DB during its own `OnModuleDestroy`. The full hook catalog and timing live in [[nestjs/fundamentals/lifecycle-hooks|Lifecycle hooks]].

### Dynamic modules are no longer deduplicated by hash

```typescript
import { Module } from "@nestjs/common";
import { TypeOrmModule } from "@nestjs/typeorm";
import { User } from "./user.entity";

// v10: these two imports were merged into one module instance.
// v11: they are two separate instances.
@Module({
  imports: [TypeOrmModule.forFeature([User]), TypeOrmModule.forFeature([User])],
})
export class BrokenModule {}

// v11 fix: hold a reference and reuse it.
const userTypeOrm = TypeOrmModule.forFeature([User]);

@Module({
  imports: [userTypeOrm, userTypeOrm],
})
export class FixedModule {}
```

Mostly invisible at runtime, but **integration tests** that stub a provider on a dynamic module can fail with "two instances, only one stubbed". Fix options (per the [migration guide](https://docs.nestjs.com/migration-guide#module-resolution-algorithm)):

- Share the module reference (above).
- `module.select(ParentModule).get(Target)` to address one instance.
- `module.get(Target, { each: true })` to grab all of them.
- Opt out: `Test.createTestingModule({}, { moduleIdGeneratorAlgorithm: "deep-hash" })` (the option is declared on [`NestApplicationContextOptions`](https://github.com/nestjs/nest/blob/master/packages/common/interfaces/nest-application-context-options.interface.ts#L48-L55) with values `'deep-hash' | 'reference'`, default `'reference'`).

### Global-module middleware runs first

In v10, middleware order followed the topological sort of the module graph regardless of where the middleware was declared. In v11, [[nestjs/fundamentals/middleware|middleware]] from global modules now consistently runs **before** middleware from imported modules ([migration guide → Middleware registration order](https://github.com/nestjs/docs.nestjs.com/blob/master/content/migration.md#middleware-registration-order): "middleware registered in global modules is now executed first, regardless of its position in the module dependency graph"). If you relied on a feature module's middleware running before a global one, the order now flips.

### `Reflector#getAllAndOverride` is `T | undefined`

Type-only change but it'll show up at compile time:

```typescript
import { CanActivate, ExecutionContext, Injectable } from "@nestjs/common";
import { Reflector } from "@nestjs/core";

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    // v10: const roles: string[] = this.reflector.getAllAndOverride("roles", [
    //   context.getHandler(),
    //   context.getClass(),
    // ])

    // v11
    const roles = this.reflector.getAllAndOverride<string[]>("roles", [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!roles) return true; // handle the missing-metadata branch
    return roles.includes("admin");
  }
}
```

`getAll` and `getAllAndMerge` also picked up sharper type inference; `getAllAndMerge` now returns an object (not a single-element array) when only one entry exists and it's an object ([`Reflector#getAllAndMerge`](https://github.com/nestjs/nest/blob/master/packages/core/services/reflector.service.ts#L203-L223): when `metadataCollection.length === 1` and `isObject(value)`, it returns `value` directly).

## New features worth knowing

### `ConsoleLogger` JSON mode

```typescript
// main.ts
import { NestFactory } from "@nestjs/core";
import { ConsoleLogger } from "@nestjs/common";
import { AppModule } from "./app.module";

const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    json: true,
    colors: false, // implicit when json: true; set true for local dev
  }),
});
```

One line gets you structured logs ingestible by Loki, Datadog, CloudWatch, etc., with no Pino/Winston wrapper required. Source: [PR #14121](https://github.com/nestjs/nest/pull/14121).

### Microservices: `unwrap()`, `on()`, `status`

Transporters (NATS, Kafka, Redis, RMQ, gRPC...) expose three new APIs. NATS shown; the surface is identical across transports, only the underlying client type differs:

```typescript
// main.ts
import { NestFactory } from "@nestjs/core";
import { Transport } from "@nestjs/microservices";
import type { NatsConnection } from "nats";
import { AppModule } from "./app.module";

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  const serviceRef = app.connectMicroservice({
    transport: Transport.NATS,
    options: { servers: ["nats://localhost:4222"] },
  });

  // 1. Native client access
  const client = serviceRef.unwrap<NatsConnection>();
  console.log(client.info);

  // 2. Connection lifecycle events
  serviceRef.on("disconnect", () => console.log("nats down"));

  // 3. Reactive status stream
  serviceRef.status.subscribe((s) => console.log("status:", s));
  // emits: connecting | connected | disconnected | reconnecting (transporter-dependent)

  await app.startAllMicroservices();
  await app.listen(3000);
}
bootstrap();
```

Same API surface on injected `ClientProxy`. Source: [microservices > instance status updates](https://docs.nestjs.com/microservices/basics#instance-status-updates).

### `ParseDatePipe`

Built-in pipe that parses date strings/numbers from `@Param`/`@Query`/`@Body` into a `Date`, with the same `optional` / `default` ergonomics as `ParseIntPipe` and friends ([`ParseDatePipeOptions`](https://github.com/nestjs/nest/blob/master/packages/common/pipes/parse-date.pipe.ts#L10-L31)). Lands in `@nestjs/common`, no extra install.

### Microservice options from the DI container

Transporter options (broker URLs, credentials, TLS material) can now come through `useFactory` and pull values from `ConfigService` or any other provider. Previously the options object was static, which forced env access at module-evaluation time. Source: [PR #12622](https://github.com/nestjs/nest/pull/12622).

```typescript
// app.module.ts
import { Module } from "@nestjs/common";
import { ClientsModule, Transport } from "@nestjs/microservices";
import { ConfigModule, ConfigService } from "@nestjs/config";

@Module({
  imports: [
    ConfigModule.forRoot(),
    ClientsModule.registerAsync([
      {
        name: "NATS_SERVICE",
        imports: [ConfigModule],
        inject: [ConfigService],
        useFactory: (config: ConfigService) => ({
          transport: Transport.NATS,
          options: { servers: [config.getOrThrow<string>("NATS_URL")] },
        }),
      },
    ]),
  ],
})
export class AppModule {}
```

### `@nestjs/cqrs`: request-scoped handlers and typed commands/queries

CQRS handlers can now be `Scope.REQUEST`, and `Command<R>` / `Query<R>` carry the return type into the bus call:

```typescript
import { Injectable, Scope } from "@nestjs/common";
import { Command, CommandBus, CommandHandler, ICommandHandler } from "@nestjs/cqrs";

export class CreateUserCommand extends Command<{ id: string }> {
  constructor(public readonly email: string) {
    super();
  }
}

// Scope.REQUEST: a fresh handler instance per request, so it can inject
// REQUEST-scoped providers (current user, trace id, tenant connection, ...).
@CommandHandler(CreateUserCommand, { scope: Scope.REQUEST })
export class CreateUserHandler implements ICommandHandler<CreateUserCommand> {
  async execute(cmd: CreateUserCommand): Promise<{ id: string }> {
    return { id: "..." };
  }
}

@Injectable()
export class UsersService {
  constructor(private readonly commandBus: CommandBus) {}

  async create(email: string) {
    // result is { id: string }, not unknown — typed via Command<R>
    const result = await this.commandBus.execute(new CreateUserCommand(email));
    return result;
  }
}
```

Full details in the [official CQRS docs](https://docs.nestjs.com/recipes/cqrs).

### `IntrinsicException`

Thrown exceptions are auto-logged by Nest's default exception handler. `IntrinsicException` (also from `@nestjs/common`) suppresses that logging; useful for "expected" thrown errors like rejected inputs or auth denials that already get logged elsewhere. Source: [PR #14182](https://github.com/nestjs/nest/pull/14182).

## Upgrade checklist

```shell
# Recommended: npm-check-updates picks up @nestjs/* + adapters together
npx npm-check-updates -u "/^@nestjs\\//" "express" "fastify"
npm install
```

Then walk this list:

1. Bump Node to 20+ in CI, Dockerfile, `engines`.
2. Grep for `forRoutes('*')`, `forRoutes('(.*)')`, and `@Get('*')`-style routes; rename wildcards.
3. If you use bracket-style query strings on Express, add `app.set('query parser', 'extended')`.
4. If you're on Fastify, audit `enableCors({ methods })`.
5. If you use `CacheModule` with a Redis store, migrate to Keyv per [[nestjs/data/caching|the caching recipe]].
6. Re-check any code that depends on `process.env` overriding `ConfigService` factory output.
7. Audit `OnModuleDestroy` handlers for ordering assumptions.
8. Re-run the test suite; expect dynamic-module test breakage if you stub.
9. Add `| undefined` handling around `reflector.getAllAndOverride` calls.
