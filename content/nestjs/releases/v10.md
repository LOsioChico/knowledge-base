> Released June 15, 2023. Smaller release than [[nestjs/releases/v11|v11]]: drops Node 12, splits `CacheModule` into a standalone package, and ships [[nestjs/recipes/swc-setup|SWC]] integration in the CLI. Most apps upgrade in an afternoon.

## At a glance

| Change                                                                              | Type     | Impact                                                                                       |
| ----------------------------------------------------------------------------------- | -------- | -------------------------------------------------------------------------------------------- |
| Node.js 16+ required                                                                | Breaking | Drop Node 12 from CI before bumping                                                          |
| `CacheModule` moved to `@nestjs/cache-manager`                                      | Breaking | Re-import; see [[nestjs/data/caching\|the caching recipe]]                                   |
| CLI Plugins require TypeScript 4.8+                                                 | Breaking | Bump `typescript` if `@nestjs/swagger` or `@nestjs/graphql` plugins are enabled              |
| `ES2021` is the new compile target                                                  | Behavior | Smaller bundles, no polyfill ship                                                            |
| SWC compiler in `nest start -b swc`                                                 | Feature  | ~20× faster dev builds; opt-in                                                               |
| `Test#overrideModule().useModule()`                                                 | Feature  | Mock a whole module instead of every provider                                                |
| Redis microservice `wildcards: true`                                                | Feature  | Glob-pattern subscriptions via `psubscribe`                                                  |
| `AsyncLocalStorage` works inside [[nestjs/fundamentals/interceptors\|interceptors]] | Feature  | Trace context survives the response stream; see [[nestjs/recipes/trace-id\|trace-id recipe]] |
| `@Search()` HTTP method decorator                                                   | Feature  | First-class handler for `SEARCH`                                                             |
| [[nestjs/fundamentals/middleware\|Middleware]] runs once per matched route          | Behavior | Stops double-firing when multiple routes share a path                                        |

## Breaking changes you'll actually hit

### Node.js 16+

NestJS 10 drops Node 12 ([EOL April 2022](https://nodejs.org/en/about/previous-releases)). The minimum is Node 16, and every official `@nestjs/*` package now compiles to `ES2021` instead of shipping polyfills. Smaller libraries, occasionally faster, but nothing to do beyond bumping your runtime:

```json
// package.json
"engines": { "node": ">=16" }
```

By the time you read this, Node 16 is also EOL ([September 11, 2023 per the Node release schedule](https://github.com/nodejs/Release/blob/main/README.md#end-of-life-releases)). If you're upgrading fresh in 2024+, jump straight to Node 20 or 22, and read the [[nestjs/releases/v11|v11 notes]] next: v11's [README](https://github.com/nestjs/nest/blob/master/Readme.md) and the [v11 release blog post](https://trilon.io/blog/announcing-nestjs-11-whats-new) bump the minimum to Node 20.

### `CacheModule` moved out of `@nestjs/common`

`CacheModule`, `CacheInterceptor`, and `Cache` are no longer exported from `@nestjs/common`. They live in their own package now (verified against [`@nestjs/cache-manager`'s entry barrel](https://github.com/nestjs/cache-manager/blob/master/lib/index.ts) and [`@nestjs/common`'s public exports](https://github.com/nestjs/nest/tree/master/packages/common), which no longer expose them):

```shell
npm i @nestjs/cache-manager cache-manager
```

```typescript
// Before (v9)
import { CacheModule, CacheInterceptor } from "@nestjs/common";

// After (v10)
import { CacheModule, CacheInterceptor } from "@nestjs/cache-manager";
```

Within v10 the surface stays small: the same `CacheModule.register({...})` shape, same `Cache` `get`/`set`/`del` methods, same store packages. The next major (v11) swaps the store contract to [Keyv](https://keyv.org), which changes both wire format and store packages: covered in the [[nestjs/data/caching|caching recipe]] and the [[nestjs/releases/v11|v11 notes]].

### CLI Plugins require TypeScript 4.8+

`@nestjs/swagger` and `@nestjs/graphql` use the TypeScript AST to auto-generate OpenAPI / GraphQL schemas. [TypeScript 4.8 reshuffled the AST around decorators](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-4-8.html#decorators-are-placed-on-modifiers-on-typescripts-syntax-trees), so older versions can no longer be supported. Bump `typescript` to >= 4.8 (5.x is fine) before the upgrade or the plugins crash on first build.

```json
// package.json
"devDependencies": { "typescript": "^5.0.0" }
```

## New features worth knowing

### SWC compiler in the Nest CLI

`nest start` now accepts `-b swc` to swap `tsc` for [SWC](https://swc.rs), roughly **20× faster** on Nest builds. Full setup (single-app, Jest/Vitest, [[nestjs/recipes/monorepo|monorepo]] via `swc-loader`, gotchas) lives in the [[nestjs/recipes/swc-setup|SWC recipe]]: the de-facto builder for projects in this knowledge base. Quick sniff test:

```shell
npm i --save-dev @swc/core @swc/cli
nest start -b swc --type-check
```

For tests, SWC pairs with both Jest (`@swc/jest`) and Vitest. Walkthrough in the [[nestjs/recipes/swc-setup|SWC recipe]] and the [official SWC recipe](https://docs.nestjs.com/recipes/swc).

### Override whole modules in tests

Before v10, mocking a module meant overriding every provider it exposed individually. v10 adds `overrideModule().useModule()`:

```typescript
import { Test } from "@nestjs/testing";
import { LoggerModule } from "./logger.module";
import { LoggerTestingModule } from "./logger-testing.module";
import { AppModule } from "./app.module";

const moduleRef = await Test.createTestingModule({
  imports: [AppModule],
})
  .overrideModule(LoggerModule)
  .useModule(LoggerTestingModule)
  .compile();
```

The replacement module must export the same public providers as the original (same tokens, compatible types). Anywhere `AppModule` (transitively) imports `LoggerModule`, the testing version gets used instead. Source: [End-to-end testing](https://docs.nestjs.com/fundamentals/testing#end-to-end-testing).

### Redis microservice: wildcard subscriptions

Set `wildcards: true` on a Redis transporter and Nest switches from `subscribe`/`message` to `psubscribe`/`pmessage` under the hood:

```typescript
import { NestFactory } from "@nestjs/core";
import { MicroserviceOptions, Transport } from "@nestjs/microservices";
import { AppModule } from "./app.module";

const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.REDIS,
  options: {
    host: "localhost",
    port: 6379,
    wildcards: true,
  },
});
await app.listen();
```

Then `@MessagePattern("h?llo")`, `@MessagePattern("h*llo")`, and `@MessagePattern("h[ae]llo")` use Redis glob semantics ([psubscribe docs](https://redis.io/commands/psubscribe/)).

### `AsyncLocalStorage` in interceptors

v10 added explicit `AsyncLocalStorage` support across the [[nestjs/fundamentals/interceptors|interceptor]] boundary so a store opened earlier in the pipeline survives the response Observable returned by the handler ([nest#11142: "async-local-storage in interceptors"](https://github.com/nestjs/nest/pull/11142)). The [[nestjs/recipes/trace-id|trace-id recipe]] is the canonical use case.

### `@Search()` decorator

Native handler for the `SEARCH` HTTP method:

```typescript
import { Body, Controller, Search } from "@nestjs/common";

@Controller("documents")
export class DocumentsController {
  @Search()
  search(@Body() query: { q: string }) {
    return { hits: [] };
  }
}
```

Useful for [Elasticsearch-style search APIs](https://www.iana.org/assignments/http-methods/http-methods.xhtml) and any RESTful endpoint that needs a request body on a "read" verb.

### Middleware runs once per matched route

Pre-v10, when a request hit a path matching multiple controller routes (e.g., versioned endpoints), middleware registered for that path could fire multiple times. v10 deduplicates: middleware runs **once** per matched route, even if the path resolves to several handlers internally ([PR #9809](https://github.com/nestjs/nest/pull/9809)).

If you relied on the duplicate firing for side effects (rare, usually a bug), audit your middleware before bumping.

## Upgrade checklist

```shell
npx npm-check-updates -u "/^@nestjs\\//" "typescript"
npm install
```

Then walk this list:

1. Bump Node to 16+ in CI, Dockerfile, `engines`. (Strongly consider 20+ to set up for v11.)
2. Bump `typescript` to >= 4.8 if you use `@nestjs/swagger` or `@nestjs/graphql` CLI plugins.
3. Replace every `import { CacheModule, ... } from "@nestjs/common"` with `from "@nestjs/cache-manager"`. Add `@nestjs/cache-manager` and `cache-manager` to `dependencies`.
4. Re-run the test suite. Middleware-runs-once may surface latent assumptions.
5. Optional: try `nest start -b swc --type-check` for faster dev startup.
